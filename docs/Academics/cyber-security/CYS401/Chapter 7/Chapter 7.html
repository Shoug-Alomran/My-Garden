Security Architecture Principles

9 fundamental rules for designing secure systems

What is Security Architecture?

Security architecture is the overall system required to protect an organization's IT infrastructure ‚Äî including hardware, software specifications, processes, and procedures involved in preventing, mitigating, and investigating threats. Security Architecture Principles are the fundamental rules that should be applied when designing and implementing security controls.

Principle #1

üèóÔ∏è Security by Design

Security requirements should be considered from the very start of development ‚Äî not as an afterthought. Building security in early saves time, money, and effort.

üí° Tip Think of security as a foundation, not a coat of paint. Adding it later is like installing locks on a glass door.
üìù Exam Watch If a question asks "when should security requirements be addressed?", the answer is during initial design, not after development.
Principle #2

‚úÇÔ∏è Simplicity

Reduce complexity in security controls. Simpler systems have fewer mistakes, are easier to manage, and security issues are resolved faster.

üí° Tip The more complex the system, the more potential for misconfigurations. Choose simple, well-understood controls over complex ones.
üìå Example Using a single MFA solution across all systems is simpler than five different authentication methods ‚Äî and easier to audit.
Principle #3

üßÖ Defense in Depth

Use multiple layers of security. If one control fails or is bypassed, others still protect the system. No single point of failure.

üìå Real-World Example An enterprise network uses: perimeter firewall ‚Üí IDS ‚Üí network segmentation ‚Üí secure endpoints ‚Üí encryption. If the firewall is bypassed, the IDS and other layers still block the attacker.
üìù Exam Watch "Layered security" = Defense in Depth. Look for answers mentioning multiple controls working together.
Principle #4

üîë Least Privilege

Grant only the minimum privileges needed for a user or process to do their job. No more, no less.

üìå Example A database admin should only access their assigned databases ‚Äî not application servers or HR files unrelated to their role. A read-only user should not have write or delete permissions.
üìù Exam Watch Least privilege is the most tested principle. Any scenario about "giving minimum access" maps here.
Principle #5

üö´ Default Deny

The default setting is to deny all access. Access must be explicitly granted through configuration. Nothing is allowed unless specifically permitted.

üí° Tip Think of a bouncer at a club: you're not getting in unless your name is on the list. The list is empty by default.
üìù Exam Watch Also called "Implicit Deny" in firewall/ACL contexts. If no rule matches, access is denied.
Principle #6

üîí Fail Secure

When a system fails, it should fail in a secure state ‚Äî blocking access to protect data integrity, even if it causes a denial of service.

ModeResultRisk
Fail SecureAccess blockedDenial of service
Fail SafeControl disabledIllegitimate access
üìù Exam Watch Fail Secure = security priority (blocks access). Fail Safe = availability priority (allows access). Security exams favor Fail Secure.
Principle #7

üë• Separation of Duties

No single person has full control over an entire transaction. Tasks are split or shadowed between multiple people to prevent fraud.

Task Shadowing Task Splitting
üìå Example In a bank: Person A initiates a wire transfer, Person B must authorize it. Fraud requires both people to collude.
üìù Exam Watch SoD prevents any single employee from committing and concealing fraud. It's also a key control against insider threats.
Principle #8

üåê Don't Trust External Systems

External systems are not under your organization's control. Treat them as insecure by default until trust is explicitly established through verification.

üí° Tip Always validate, sanitize, and authenticate data from external APIs, partner networks, and third-party services ‚Äî never assume they are safe.
üìå Example A hospital receiving patient data from an external clinic should encrypt the connection and validate the data before importing it into their systems.
Principle #9

üõ°Ô∏è Zero Trust

No actor, system, or service ‚Äî inside or outside the network ‚Äî is trusted by default. Every interaction must be verified before access is granted.

üìå Example An employee inside the corporate office still needs MFA + device verification to access the company file system. Being on the internal network is not enough.
üìù Exam Watch Zero Trust is the evolution of "don't trust external systems" ‚Äî extended to internal systems too. Key phrase: "never trust, always verify."

‚è≥ Security is a Process ‚Äî Not a Product

Every system has vulnerabilities. The goal is assurance, not perfection. Systems, requirements, and threat contexts all change over time. Secure systems need constant maintenance:

Check for obsolete users Update antivirus signatures Patch security holes Test firewalls regularly Review access controls

Common Architecture Flaws

Attack types that exploit design and coding weaknesses

üì° Covert Channels

A covert channel is an attack technique used to transfer information in a secretive, unauthorized, or illicit manner. It can be used to extract information from ‚Äî or implant information into ‚Äî an organization without detection.

‚è±Ô∏è Covert Timing Channel One process accesses the data of another process through a shared resource such as RAM or CPU. The timing of operations leaks information.
üíæ Covert Storage Channel A process reads or writes to storage media to secretly communicate. Example: Android apps that request photo storage access may read metadata they shouldn't.

ü™§ Maintenance Hook (Backdoor)

A trapdoor in software that allows developers easy maintenance access ‚Äî bypassing normal authentication. Left in production, it becomes a serious vulnerability.

üìù Exam Watch If an exam scenario describes hidden developer access that skips normal checks ‚Üí Maintenance Hook / Backdoor.

üßÇ Salami (Aggregation) Attack

Small attacks that individually go undetected, but add up to one major attack. Also called an incremental attack.

üìå Example A bank employee rounds down every transaction to the nearest cent and redirects those fractions to their account. Each transaction is negligible; over millions it becomes huge.

‚úèÔ∏è Data Diddling

Changing data before or during entry into a computer system. The data is altered at the input stage, making it hard to detect after the fact.

üìå Example A payroll clerk changes an employee's hours from 40 to 400 before submitting the timesheet to the payroll system.

üëë Privileged Programs

Programs that can grant users extra privileges beyond what they are already assigned ‚Äî creating a pathway for privilege escalation attacks.

üìù Exam Watch Privileged programs violate least privilege. An attacker who exploits one can gain elevated access across the system.

üõ†Ô∏è How to Protect Against These Flaws

Trusted Recovery
Mechanisms to ensure that failures don't compromise secure operation. The system returns to a known-good secure state after a failure.
Input Validation
Validating and sanitizing all input before it is processed prevents data diddling, injection attacks, and parameter manipulation.

CIA Techniques

Ensuring Confidentiality, Integrity, and Availability

The CIA Triad in Software Design

Software designers use techniques to ensure programs do only what is required and nothing more. When controls fail: affected programs compromise Confidentiality ‚Üí unpredictable data changes break Integrity ‚Üí data corruption causes loss of Availability.

üîí Confinement Confidentiality

Process confinement restricts a process to reading from and writing to only specific memory locations and resources. Used in operating systems for process isolation and memory protection.

üìå Example ‚Äî Sandboxing Sandboxie runs programs in an isolated environment so that any changes or malware cannot affect the host system or other programs. Web browsers use sandboxing to isolate tabs from each other and from the OS.
üí° Tip Confinement = the concept. Sandboxing = a popular implementation of confinement.

üî¨ Process Isolation Integrity

Process isolation ensures that the behavior of one process will only affect its own memory and resources. Separate processes cannot interfere with each other.

Each process gets its own virtual memory space. Page tables are disjoint ‚Äî no process can access another's memory pages. Virtual addresses for each process start from 0, but map to different physical locations.

üìù Exam Watch Process isolation is the OS mechanism that prevents a compromised process from reading another process's sensitive data (e.g., passwords in memory).

‚≠ï Bounds (Authority Levels) Access Control

Each process running on a system is assigned an authority level that tells the OS what the process is allowed to do. The ring model defines privilege levels:

Ring 0 ‚Äî Most Privileged Kernel ‚Äî Direct hardware access
Rings 1 & 2 Device Drivers ‚Äî Hardware management
Ring 3 ‚Äî Least Privileged Applications ‚Äî User programs
üìù Exam Watch The two most commonly tested authority levels are User mode (Ring 3) and Kernel mode (Ring 0). Kernel mode = full access. User mode = restricted access.

Bell-LaPadula vs. Biba

The two most tested security models

üîµ Bell-LaPadula Model Confidentiality

Designed to protect confidentiality in military-style clearance systems. Uses a linear ordering of security levels.

LevelSubjectObject
Top SecretTamaraPersonnel Files
SecretSamuelE-Mail Files
ConfidentialClaireActivity Logs
UnclassifiedAliceTelephone Lists
Two Core Rules No Read Up ‚Äî A subject cannot read objects at a higher level than their clearance.

No Write Down ‚Äî A subject cannot write data to a lower classification level (prevents leaking secrets).
üìå Examples ‚úÖ Tamara (Top Secret) can read ALL files.
‚ùå Claire (Confidential) CANNOT read Personnel or E-Mail Files.
‚úÖ Alice (Unclassified) can only read Telephone Lists.
üìù Exam Watch Bell-LaPadula = Confidentiality. "No Read Up, No Write Down." Military clearances. Remember: it does NOT protect integrity.

üü¢ Biba Model Integrity

Designed to protect data integrity. The opposite philosophy of Bell-LaPadula ‚Äî focuses on preventing corruption of higher-integrity data.

Two Core Rules No Write Up ‚Äî A subject cannot write to a higher integrity level than their own (can't corrupt trusted data).

No Read Down ‚Äî A subject cannot read from a lower integrity level (prevents contamination by untrustworthy data).
üìå Example A Secret-level analyst cannot write changes to Top Secret documents (No Write Up). They also cannot read Unclassified internet sources and incorporate them into Secret reports (No Read Down ‚Äî data contamination risk).
üìù Exam Watch Biba = Integrity. "No Write Up, No Read Down." It's the reverse of Bell-LaPadula's reading direction. Biba does NOT protect confidentiality.

Access Control Matrix

Mapping subjects, objects, and their access rights

An Access Control Matrix is a table that maps subjects (users) to objects (resources) and the rights each subject has over each object.

Columns ‚Üí ACLs (tied to objects) Rows ‚Üí Capability Lists (tied to subjects)

üìå Reading the Matrix If you look at Column "File 1" (the ACL): John (Own,R,W), Alice (R), Bob (R,W) ‚Äî these are all the subjects with access to File 1.

If you look at Row "Alice" (her Capability List): File 1 (R), File 2 (Own,R,W), File 3 (W), File 4 (R), Account 1 (Inquiry, Debit), Account 2 (Inquiry, Credit) ‚Äî these are all the things Alice can do.
üìù Exam Watch ACL = per-object view (who can access this file?). Capability List = per-subject view (what can this user access?). Both come from the same matrix ‚Äî just read in different directions.

Take-Grant Model

Tracking how rights propagate through a system

The Take-Grant model dictates how rights can be passed between subjects. It's used to determine when rights can change and where privilege leakage occurs.

RuleDescriptionNotation
Take Rule Allows a subject to take rights over an object from another subject "x takes (Œ± to z) from y"
Grant Rule Allows a subject to grant rights to an object to another subject "x grants (Œ± to z) to y"
Create Rule Allows a subject to create new rights "x creates (Œ± to) new {subject/object} n"
Remove Rule Allows a subject to remove rights it holds "x removes (Œ± to) y"
üí° Key Insight The Take-Grant model lets security analysts determine when rights in a system can change and identify where privilege leakage might occur ‚Äî i.e., how an unauthorized party could gain access rights through a chain of grants and takes.
üìå Example of Leakage Alice grants Bob the "read" right on File X. Bob then takes that right and grants it to Charlie (who was not authorized). Through a chain of Take-Grant operations, Charlie now has unauthorized access ‚Äî this is privilege leakage.
üìù Exam Watch Take-Grant is about tracing permission propagation. Exam questions might ask you to follow a chain of rules to determine if a subject can gain access to a resource they weren't originally authorized for.
