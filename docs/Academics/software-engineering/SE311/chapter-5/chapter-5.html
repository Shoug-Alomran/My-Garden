1. Requirements Analysis Overview

🎯 What is Requirements Analysis?

Requirements Analysis is the process of understanding the application domain, identifying user needs, capturing system requirements, and validating them.

Main Goals

  • Understand the application domain and identify user needs
  • Collect data on system requirements and system constraints
  • Define development effort scope and system design goals
  • Determine the risks of developing the system:
    • Economic risks
    • Technical risks
    • Operational risks
    • Organizational risks
    • Legal risks
  • Capture the system requirements:
    • Data requirements → Domain Model
    • Functional requirements → Use-Case Model
    • Nonfunctional requirements → Supplementary text
  • Validate the system requirements - verify correctness and completeness

3. System Requirements Specification (SRS)

📄 What is an SRS?

The System Requirements Specification (SRS) documents the system requirements. It is the official statement of what is required of the system.

Key Characteristics of SRS

  • Should include both a definition of user requirements and a specification of system requirements
  • It is NOT a design document
  • Should state WHAT the system should do, but not HOW it should do it

⚠️ Agile Perspective

Many agile methods argue that producing a formal SRS is a waste of time as requirements change so quickly. However, some level of requirements documentation is still valuable for communication and traceability.

Ways of Writing an SRS

Method Description Advantages Disadvantages
Natural Language Uses sentences supplemented by diagrams and tables Easy to write, universally understood Ambiguous, imprecise
Structured Natural Language Uses restricted language following a fixed standard/template More consistent, less ambiguous Can be rigid, requires discipline
Graphical Notations (UML) Uses graphical models plus text annotations Visual, precise, standard notation Requires training, can be complex
Design Description Languages Uses a language like a programming language Very precise, executable Technical, not user-friendly
Mathematical Specifications Uses notations based on mathematical concepts Unambiguous, provable Difficult to understand, time-consuming

4. Domain Modeling

Definition & Purpose

🎯 What is Domain Modeling?

Domain Modeling captures the most important classes and their associations for which data must be stored.

  • Focuses on things that exist or events that occur
  • Described in a class diagram
  • Provides a glossary of terms

Sources of Domain Model Elements

  • Requirements statements - written specifications
  • Domain experts (users) - interviews and discussions
  • Application documentation - existing systems and manuals

Types of Classes

📦 Class Categories

  • Business Objects: orders, accounts, invoices, contracts
  • Real-World Objects and Concepts: suppliers, customers, employees, products
  • Events: aircraft arrival/departure, sales, reservations, purchases, rentals

💡 Key Principle

Identify only relevant classes and associations - those that are essential and will always exist; are not transient. This leads to a stable system.

⚠️ Important Note

The decomposition of user requirements into classes and associations depends on judgment and experience and the nature of the problem. There is usually no one correct decomposition!

Identifying Classes and Associations

🔍 How to Identify Classes

Look for naturally occurring things or concepts in the user requirements:

  • Classes appear as nouns or noun phrases
  • Associations appear as verbs or verb phrases

✅ Best Practices

  • Put all terms into singular form (Movie, not Movies)
  • Use active voice for associations
  • Identify only relevant classes/associations
  • Focus on those that are essential and will always exist
  • Avoid transient or temporary concepts

📝 Example: Finding Classes from Text

"The system must be able to record which movies are sold and rented and by whom (customer)."

Identified Classes:

  • Movie - noun phrase
  • Customer - noun implied by "by whom"

Identified Associations:

  • Customer purchases Movie - "sold" indicates purchase relationship
  • Customer rents Movie - "rented" indicates rental relationship

Identifying Associations

🔗 Association Identification Rules

Associations represent structural relationships between classes:

  • Look for verbs connecting nouns (e.g., "customer rents movie")
  • Associations should describe persistent relationships, not transient events
  • Should answer: "What relationships exist between these classes?"
Requirement Text Classes Association
"Members can make reservations for physical movie rentals" Member, RentalCopy Member reserves RentalCopy
"Movie has rental copies" Movie, RentalCopy Movie HasPhysical RentalCopy
"Customers provide reviews for movies" Customer, Review, Movie Customer Provides Review, Review IsFor Movie

Identifying Attributes

📊 How to Find Attributes

  • Attributes usually correspond to nouns followed by possessive phrases
    Example: "password of student", "student's address"
  • Adjectives usually represent specific enumerated values
    Example: "Fall term", "PhD degree"
  • Only identify attributes that directly relate to the application domain

⚠️ Important Note

Most attributes will NOT be given in a requirements statement and will have to be obtained from domain experts or application documentation.

📝 Example: Extracting Attributes

Evaluating Classes

✔️ Class Evaluation Checklist

Ask these questions about each identified class:

Question Action if YES Reason
Is the class irrelevant to the domain model? Eliminate Only include classes that represent persistent data needed by the system
Is the class vague (ill-defined)? Make specific or eliminate A class should be specific and well-defined
Are there redundant classes? Keep most descriptive Avoid duplication and confusion
Should the class really be an attribute? Consider requirements Ask: Is the concept dependent or independent?
Does the class name describe a role? Be careful Roles also appear as nouns but may not be classes
Does the class describe an operation/action? Eliminate Classes represent things, not actions
Does the class describe implementation constructs? Eliminate Domain model is implementation-independent

⚠️ Critical Rule: Persistent vs. Transient Data

A class should represent persistent data, not transient data.

Persistent: Data that needs to be stored in files or database (e.g., Customer, Movie, Order)

Transient: Temporary data during execution (e.g., SearchResults, ValidationStatus)

Evaluating Associations

🔗 Association Evaluation Checklist

Question Action if YES Reason
Is the association irrelevant to the domain model? Eliminate Only include necessary relationships
Does the association describe an operation? Eliminate Should describe structural property, not transient event
Can ternary associations be decomposed into binary? Decompose Keep model as simple as possible
Is it a derived association? Eliminate Introduces redundancy - can be computed from other associations
Does it describe implementation constructs? Eliminate Domain model is implementation-independent

📝 Example: Derived Association

    A ────X──── B ────Y──── C
    
    If association X means "A relates to B"
    And association Y means "B relates to C"
    
    Then we can DERIVE: A relates to C (through B)
    
    Therefore, if there's an association Z: A ────Z──── C
    It's a DERIVED association and should be eliminated!

Why? Because Z is redundant - we can find the relationship between A and C by following X then Y.

Evaluating Attributes

📊 Attribute Evaluation Checklist

Question What to Check Reason
Are attributes closely related to their class? Each attribute should logically belong to its class A class should be simple and coherent
Should any attributes really be classes? Is the concept dependent on the class? If independent and needs multiple attributes, make it a class
Should any attributes be in association class? Does attribute depend on existence of association? Quantity sold depends on Customer-Movie purchase association
Have object identifiers been included? Check for ID fields used for relationships Object identifiers should NOT be in domain model!

❌ Common Mistake: Object Identifiers as Attributes

WRONG:

Customer
- customerID
- movieID  ← This is an object identifier pointing to Movie!

CORRECT:

Customer ────purchases───→ Movie

Why? In domain modeling, relationships are shown through associations, not through ID attributes. IDs are implementation details!

Specifying Details

📋 Complete Domain Model Specification

For Each Class:

  • Specify its attributes
  • Each class is assumed to have two standard operations: get and set (don't need to specify)

For Each Attribute:

  • Name - must be unique within a class
  • Type - string, integer, date, boolean, etc.
  • Multiplicity - if greater than 1 (e.g., leadingActor[0..*])

For Each Association:

  • Name - should say "what" not "how" or "why"
  • Role names - if needed for clarity
  • Multiplicity - if known (use "?" if cannot be inferred)
  • Association class - if needed

5. Complete Movie Shop Case Study

Requirements Statement

🎬 Movie Shop System Requirements

The following are the requirements for a web-based system to computerize the management of the sale and rental of movies for a movie shop:

  1. The system must be able to handle both physical and digital movies.
  2. It must be able to record which movies are sold and rented and by whom.
  3. For sold movies, the quantity sold should be recorded; for physical movie rental, which copy is rented and when it is due back should be recorded.
  4. The system should keep track of overdue rentals of physical movies and send email notices to customers who have movies overdue.
  5. There will be a customer membership option for an annual fee, which will entitle a member to discounts (10%) on the sale and rental of movies.
  6. Members should be able to make reservations for physical movie rentals either in person at the shop, by telephone or via the Web.
  7. A member can reserve at most five physical movies at any one time, but there is no limit on how many physical movies a member or nonmember can rent at any one time.
  8. As an added feature, the shop would like to allow customers (either members or nonmembers) to input, via the Web, mini-reviews (up to 100 words) and a rating (from 1, lowest, to 10, highest) of movies they have purchased or rented.
  9. These reviews should be anonymous if the customer so wishes (i.e., customers can specify whether they want their name to be made known when other customers browse the reviews).
  10. A sales clerk should be able to enter and update the following information about all customers (members or nonmembers): name, address, phone number, age, sex, and email address.
  11. Members are assigned a membership number by the shop when they become members and a password, which allows them to change their personal information and to buy and rent digital movies via the Web.
  12. The shop manager should be able to generate various reports on the sale and rental of movies.
  13. A sales clerk should be able to sell and rent physical movies and process the return of rented physical movies.
  14. When selling or renting physical movies, a sales clerk must be able to look up customer information and determine whether the customer is a member.
  15. A sales clerk must be able to enter basic information about a movie (i.e., movie id, title, leading actor(s), director, producer, genre, synopsis, release year, running time, selling price, and rental price).

Detailed Requirements Analysis

🔍 Step-by-Step Analysis Process

We analyze each requirement to extract classes, attributes, associations, and constraints:

Requirement 1: Physical and Digital Movies

"The system must be able to handle both physical and digital movies."

Analysis:

  • Classes: Movie
  • Attributes:
    • Movie: isPhysical (Boolean)
    • Movie: isDigital (Boolean)

Reasoning: A movie can be both physical and digital, so we use boolean flags rather than creating separate subclasses.

Requirement 2: Record Sales and Rentals

"It must be able to record which movies are sold and rented and by whom."

Analysis:

  • Classes & Associations:
    • Customer Purchases Movie
    • Customer Rents Movie

Reasoning: "by whom" implies a Customer class, and the verbs "sold" and "rented" create associations.

Requirement 3: Quantity and Copy Details

"For sold movies, the quantity sold should be recorded; for physical movie rental, which copy is rented and when it is due back should be recorded."

Analysis:

  • Classes: RentalCopy
  • Associations:
    • Movie HasPhysical RentalCopy
    • Customer RentsPhysical RentalCopy
  • Attributes:
    • Purchases: quantity (Integer) - association class attribute
    • RentalCopy: copyNumber (String)
    • RentalCopy: returnDate (Date)

Reasoning: We need a RentalCopy class to track individual physical copies. Quantity belongs to the Purchase association.

Requirement 4: Overdue Tracking

"The system should keep track of overdue rentals of physical movies and send email notices to customers who have movies overdue."

Analysis:

  • Attributes: Customer: email (String)

Reasoning: Email is needed for notification. Overdue status can be derived from returnDate.

Requirement 5: Membership and Discounts

"There will be a customer membership option for an annual fee, which will entitle a member to discounts (10%) on the sale and rental of movies."

Analysis:

  • Generalization: Member is a kind of Customer
    • Customer Generalizes Member

Reasoning: Member is a specialized type of Customer with additional attributes and privileges.

Requirement 6: Reservations

"Members should be able to make reservations for physical movie rentals either in person at the shop, by telephone or via the Web."

Analysis:

  • Association: Member Reserves RentalCopy

Reasoning: Only members can make reservations. The method (in person, phone, web) is not part of the domain model.

Requirement 7: Reservation and Rental Limits

"A member can reserve at most five physical movies at any one time, but there is no limit on how many physical movies a member or nonmember can rent at any one time."

Analysis:

  • Constraints:
    • max-card(Member, Reserves) = 5
    • max-card(Customer, RentsPhysical) = *

Reasoning: Multiplicity constraints capture business rules about quantity limits.

Requirement 8: Reviews and Ratings

"As an added feature, the shop would like to allow customers (either members or nonmembers) to input, via the Web, mini-reviews (up to 100 words) and a rating (from 1, lowest, to 10, highest) of movies they have purchased or rented."

Analysis:

  • Classes: Review
  • Associations:
    • Customer Provides Review
    • Review IsFor Movie
  • Attributes:
    • Review: reviewText (String)
    • Review: rating (Integer)

Reasoning: Review is a separate class because it has multiple attributes and relates two other classes.

Requirement 9: Anonymous Reviews

"These reviews should be anonymous if the customer so wishes (i.e., customers can specify whether they want their name to be made known when other customers browse the reviews)."

Analysis:

  • Attributes: Review: anonymous (Boolean)

Reasoning: A flag to indicate if the review should display the customer's name.

Requirement 10: Customer Information

"A sales clerk should be able to enter and update the following information about all customers (members or nonmembers): name, address, phone number, age, sex, and email address."

Analysis:

  • Attributes:
    • Customer: name (String)
    • Customer: address (String)
    • Customer: phoneNumber (String)
    • Customer: age (Integer)
    • Customer: sex (String)
    • Customer: email (String)

Reasoning: Direct mapping from requirements to attributes.

Requirement 11: Member Credentials

"Members are assigned a membership number by the shop when they become members and a password, which allows them to change their personal information and to buy and rent digital movies via the Web."

Analysis:

  • Attributes:
    • Member: memberNumber (String)
    • Member: password (String)

Reasoning: These attributes are specific to Member, not all Customers.

Requirements 12-14: Functional Requirements

"The shop manager should be able to generate various reports..."

"A sales clerk should be able to sell and rent physical movies..."

"When selling or renting physical movies, a sales clerk must be able to look up customer information..."

Analysis:

  • No new domain model requirements

Reasoning: These are functional requirements describing what users can DO, not data that needs to be stored.

Requirement 15: Movie Information

"A sales clerk must be able to enter basic information about a movie (i.e., movie id, title, leading actor(s), director, producer, genre, synopsis, release year, running time, selling price, and rental price)."

Analysis:

  • Attributes:
    • Movie: movieId (String)
    • Movie: title (String)
    • Movie: leadingActor[0..*] (String) - multiple actors possible
    • Movie: director (String)
    • Movie: producer (String)
    • Movie: genre (String)
    • Movie: synopsis (String)
    • Movie: releaseYear (Integer)
    • Movie: runningTime (Integer)
    • Movie: sellingPrice (Real)
    • Movie: rentalPrice (Real)

Reasoning: Note that leadingActor has multiplicity [0..*] because a movie can have multiple leading actors.

Alternative Domain Model Solutions

💡 Important: Multiple Valid Solutions

There are several valid ways to model purchases and rentals. Here are three common approaches:

Solution 1: Direct Associations

Customer ─────Purchases────→ Movie
         (quantity)
         
Customer ─────Rents────→ Movie

Constraint: If Purchases is an association class, 
then a customer can purchase a given movie at most once.

Pros: Simple, direct

Cons: Cannot track multiple purchases of same movie, cannot track rental history

Solution 2: Transaction Classes (Preferred)

Customer ─────Has────→ Purchase ─────HasPurchase────→ Movie
                       (quantity)

Customer ─────Has────→ Rental ─────HasRental────→ Movie

Advantage: Can track multiple transactions over time
Can store transaction dates, amounts, etc.

Pros: More flexible, tracks history, allows multiple purchases

Cons: More complex

Solution 3: Generalized Transaction

Customer ─────HasTransaction────→ Transaction ─────Has────→ Movie
                                       ▲
                                       │
                          ┌────────────┴────────────┐
                          │                         │
                      Purchase                   Rental

Advantage: Most general, allows easy addition of new transaction types

Pros: Extensible, unified handling of transactions

Cons: Most complex, may be over-engineered

⚠️ Important Distinction

Movie vs. Physical Copy

An instance of the class Movie does not represent an instance of a physical copy of the movie, but a description of the movie.

Hence, the same instance of Movie (i.e., the description) can be related to many rental copies, rentals, and purchases.

6. Common Domain Modeling Errors

❌ Common Mistakes to Avoid

1. Using IDs to Relate Classes

WRONG:

Customer
- customerID
- movieID  ← Don't use IDs to link classes!

CORRECT:

Customer ────association───→ Movie

Why? Object identifiers should NOT be included as attributes in the domain model! Use associations instead.

2. Representing the Client/Organization

WRONG:

┌──────────┐
│   Shop   │
└──────────┘

Why? The "Shop" is the organization running the system, not data that needs to be stored. Don't include it in the domain model.

3. Incorrect Use of Generalization

WRONG:

      Person
        △
        │
    ┌───┴────┐
Customer  Nonmember

PhysicalMovie  DigitalMovie
        △          △
        └────┬─────┘
           Movie

Why?

  • Person is too general - not needed
  • Nonmember is just a Customer without membership - don't need separate class
  • PhysicalMovie/DigitalMovie - a movie can be BOTH, so use attributes instead

CORRECT:

    Customer
        △
        │
      Member

Movie (with isPhysical, isDigital attributes)

4. Incorrect Use of Aggregation/Composition

WRONG:

Movie ◇────── RentalCopy
Movie ◇────── Review
Customer ◇────── Member

Why? These are not part-whole relationships!

  • RentalCopy is not a "part" of Movie - it's a separate physical instance
  • Review is not a "part" of Movie - it's an independent entity
  • Member is not a "part" of Customer - it's a specialized type (use generalization)

5. Incorrect Use of Association Class

WRONG:

Customer ─────Rents────→ Movie
           (returnDate)

Problem: returnDate is for a specific COPY, not the movie in general!

CORRECT:

Customer ─────RentsPhysical────→ RentalCopy
                                 (returnDate is an attribute of RentalCopy)

6. Incorrect Constraints

WRONG:

Movie can be {XOR} Physical or Digital

Why? A movie can be BOTH physical and digital!

CORRECT: Use boolean attributes isPhysical and isDigital (both can be true)

7. Representing Operations

WRONG:

┌─────────────┐
│  Generates  │  ← This is an operation, not a class!
│   Browses   │
│   Enters    │
│  LooksUp    │
└─────────────┘

Why? These are actions/functions, not things that need to be stored.

8. Storing Reports

WRONG:

┌───────────────┐
│  SalesReport  │  ← Don't store reports as classes!
│ RentalReport  │
└───────────────┘

Why? Reports are generated from data, not persistent data themselves.

9. Representing Implementation Aspects

WRONG:

┌──────────┐
│  System  │  ← Implementation construct
│   Web    │  ← Access method
│Telephone │  ← Access method
│ Reports  │  ← Generated output
└──────────┘

Why? Domain model should be implementation-independent!

10. Over-Specifying the Model

WRONG:

┌──────────────┐
│ SalesClerk   │  ← Role, not persistent data
│   Manager    │  ← Role, not persistent data
└──────────────┘

Why? These are user roles for functional requirements, not data to store.

Key Question: What information about things/procedures needs to be persistent (i.e., in files or a database)?

💡 Golden Rule for Domain Modeling

Only include classes, associations, and attributes that represent persistent data needed by the system.

Ask yourself: "Does this information need to be stored in a database?" If no, it probably shouldn't be in the domain model.

7. Exam Tips & Strategy

📝 How to Approach Domain Modeling Questions

Step-by-Step Process

  1. 1Read the requirements carefully - multiple times if needed
  2. 2Identify all nouns → potential classes
  3. 3Identify all verbs → potential associations
  4. 4Look for possessive phrases → potential attributes
  5. 5Apply evaluation criteria to filter out invalid elements
  6. 6Determine multiplicities from requirements and domain knowledge
  7. 7Check for generalizations (is-a relationships)
  8. 8Identify association classes (attributes that depend on associations)
  9. 9Draw the diagram clearly and legibly
  10. 10Review using the common errors checklist

Time Management

  • Analysis phase (40%): List classes, associations, attributes, constraints
  • Evaluation phase (20%): Apply evaluation criteria, remove invalid elements
  • Diagram phase (30%): Draw the class diagram
  • Review phase (10%): Check for errors, verify multiplicities

Common Exam Question Types

Type 1: Complete Domain Model

"From the requirements statement, identify all classes, attributes, associations, aggregations/compositions, generalizations, and multiplicity constraints. Construct a complete class diagram."

What to do:

  • Create a comprehensive analysis table
  • List all elements systematically
  • Draw complete diagram with all multiplicities
  • Include all attributes with types

Type 2: Identify Specific Elements

"List all the classes that should be included in the domain model."

What to do:

  • Focus only on what's asked
  • Apply evaluation criteria strictly
  • Justify why each element is included

Type 3: Find Errors

"The following domain model was created for the system. Identify and explain the errors."

What to do:

  • Check against all evaluation criteria
  • Look for common errors (IDs, operations, implementation details)
  • Verify multiplicities
  • Check for missing or incorrect relationships

🎯 ما هو تحليل المتطلبات؟

تحليل المتطلبات هو عملية فهم مجال التطبيق، وتحديد احتياجات المستخدم، وجمع متطلبات النظام، والتحقق من صحتها.

الأهداف الرئيسية

  • فهم مجال التطبيق وتحديد احتياجات المستخدم
  • جمع البيانات حول متطلبات النظام وقود النظام
  • تحديد نطاق جهود التطوير وأهداف تصميم النظام
  • تحديد المخاطر لتطوير النظام:
    • المخاطر الاقتصادية
    • المخاطر التقنية
    • المخاطر التشغيلية
    • المخاطر التنظيمية
    • المخاطر القانونية
  • جمع متطلبات النظام:
    • متطلبات البيانات → نموذج النطاق
    • المتطلبات الوظيفية → نموذج حالات الاستخدام
    • المتطلبات غير الوظيفية → نص تكميلي
  • التحقق من صحة متطلبات النظام - التحقق من الصحة والاكتمال

3. مواصفات متطلبات النظام (SRS)

📄 ما هي مواصفات متطلبات النظام؟

مواصفات متطلبات النظام (SRS) توثق متطلبات النظام. وهي البيان الرسمي لما هو مطلوب من النظام.

الخصائص الرئيسية لـ SRS

  • يجب أن تتضمن كلًا من تعريف متطلبات المستخدم ومواصفات متطلبات النظام
  • هي ليست وثيقة تصميم
  • يجب أن تنص على ماذا يجب أن يفعل النظام، وليس كيف يجب أن يفعله

⚠️ منظور Agile

تجادل العديد من منهجيات Agile أن إنتاج SRS رسمي هو مضيعة للوقت لأن المتطلبات تتغير بسرعة. ومع ذلك، لا يزال هناك مستوى معين من توثيق المتطلبات قيمًا للتواصل وإمكانية التتبع.

طرق كتابة SRS

الطريقة الوصف المزايا العيوب
اللغة الطبيعية يستخدم جملًا مكملة بمخططات وجداول سهل الكتابة، مفهوم عالميًا غامض، غير دقيق
اللغة الطبيعية المنظمة يستخدم لغة مقيدة تتبع معيار/قالب ثابت أكثر اتساقًا، أقل غموضًا يمكن أن يكون صارمًا، يتطلب انضباطًا
الرموز الرسومية (UML) يستخدم نماذج رسومية بالإضافة إلى تعليقات نصية مرئي، دقيق، رمز قياسي يتطلب تدريبًا، يمكن أن يكون معقدًا
لغات وصف التصميم يستخدم لغة مثل لغة البرمجة دقيق جدًا، قابل للتنفيذ تقني، ليس صديقًا للمستخدم
المواصفات الرياضية يستخدم رموزًا تعتمد على مفاهيم رياضية غير غامض، قابل للإثبات صعب الفهم، يستغرق وقتًا طويلاً

4. نمذجة النطاق

التعريف والغرض

🎯 ما هي نمذجة النطاق؟

نمذجة النطاق تلتقط أهم الفئات وارتباطاتها التي يجب تخزين بياناتها.

  • تركز على الأشياء الموجودة أو الأحداث التي تحدث
  • توصف في مخطط الفئات
  • توفر قاموس المصطلحات

مصادر عناصر نموذج النطاق

  • بيانات المتطلبات - المواصفات المكتوبة
  • خبراء المجال (المستخدمون) - المقابلات والمناقشات
  • وثائق التطبيق - الأنظمة والأدلة الحالية

أنواع الفئات

📦 فئات الفئات

  • كائنات الأعمال: الطلبات، الحسابات، الفواتير، العقود
  • الكائنات والمفاهيم الواقعية: الموردون، العملاء، الموظفون، المنتجات
  • الأحداث: وصول/مغادرة الطائرات، المبيعات، الحجوزات، المشتريات، التأجيرات

💡 المبدأ الرئيسي

حدد فقط الفئات والارتباطات ذات الصلة - تلك الأساسية وستكون موجودة دائمًا؛ ليست عابرة. هذا يؤدي إلى نظام مستقر.

⚠️ ملاحظة مهمة

يعتمد تحليل متطلبات المستخدم إلى فئات وارتباطات على الحكم والخبرة وطبيعة المشكلة. عادة لا يوجد تحليل واحد صحيح!

تحديد الفئات والارتباطات

🔍 كيفية تحديد الفئات

ابحث عن الأشياء أو المفاهيم التي تحدث بشكل طبيعي في متطلبات المستخدم:

  • الفئات تظهر كـ أسماء أو عبارات اسمية
  • الارتباطات تظهر كـ أفعال أو عبارات فعلية

✅ أفضل الممارسات

  • ضع جميع المصطلحات في صيغة المفرد (فيلم، وليس أفلام)
  • استخدم الصيغة الفعالة للارتباطات
  • حدد فقط الفئات/الارتباطات ذات الصلة
  • ركز على تلك الأساسية وستكون موجودة دائمًا
  • تجنب المفاهيم العابرة أو المؤقتة

📝 مثال: إيجاد الفئات من النص

"يجب أن يكون النظام قادرًا على تسجيل أي أفلام يتم بيعها وتأجيرها ومن قبل من (عميل)."

الفئات المحددة:

  • فيلم - عبارة اسمية
  • عميل - اسم ضمني بـ"من قبل من"

الارتباطات المحددة:

  • العميل يشتري فيلم - "باعت" يشير إلى علاقة الشراء
  • العميل يؤجر فيلم - "أجرت" يشير إلى علاقة التأجير

تحديد الارتباطات

🔗 قواعد تحديد الارتباطات

تمثل الارتباطات العلاقات الهيكلية بين الفئات:

  • ابحث عن الأفعال التي تربط الأسماء (مثل "العميل يؤجر فيلم")
  • يجب أن تصف الارتباطات علاقات دائمة، وليست أحداثًا عابرة
  • يجب أن تجيب: "ما هي العلاقات الموجودة بين هذه الفئات؟"
نص المتطلب الفئات الارتباط
"يمكن للأعضاء إجراء حجوزات لتأجير الأفلام المادية" عضو، نسخة تأجير العضو يحجز نسخة تأجير
"الفيلم له نسخ تأجير" فيلم، نسخة تأجير الفيلم له نسخة تأجير مادية
"يقدم العملاء تقييمات للأفلام" عميل، تقييم، فيلم العميل يقدم تقييم، التقييم خاص بالفيلم

تحديد السمات

📊 كيفية إيجاد السمات

  • السمات تتوافق عادة مع أسماء تليها عبارات ملكية
    مثال: "كلمة مرور الطالب"، "عنوان الطالب"
  • الصفات تمثل عادة قيمًا محددة معدودة
    مثال: "فصل الخريف"، "درجة الدكتوراه"
  • حدد فقط السمات التي ترتبط مباشرة بمجال التطبيق

⚠️ ملاحظة مهمة

معظم السمات لن تُعطى في بيان المتطلبات وسيتعين الحصول عليها من خبراء المجال أو وثائق التطبيق.

"يجب أن يكون كاتب المبيعات قادرًا على إدخال وتحديث المعلومات التالية حول جميع العملاء: الاسم، العنوان، رقم الهاتف، العمر، الجنس، وعنوان البريد الإلكتروني."

  • العنوان

تقييم الفئات

✔️ قائمة مراجعة تقييم الفئات

اطرح هذه الأسئلة حول كل فئة محددة:

السؤال الإجراء إذا كانت الإجابة نعم السبب
هل الفئة غير ذات صلة بنموذج النطاق؟ احذف قم بتضمين الفئات التي تمثل بيانات دائمة يحتاجها النظام فقط
هل الفئة غامضة (غير محددة جيدًا)؟ اجعلها محددة أو احذف يجب أن تكون الفئة محددة ومحددة جيدًا
هل هناك فئات مكررة؟ احتفظ بالأكثر وصفية تجنب التكرار والالتباس
هل يجب أن تكون الفئة حقًا سمة؟ فكر في المتطلبات اسأل: هل المفهوم تابع أم مستقل؟
هل يصف اسم الفئة دورًا؟ كن حذرًا تظهر الأدوار أيضًا كأسماء ولكن قد لا تكون فئات
هل تصف الفئة عملية/إجراء؟ احذف تمثل الفئات أشياء، وليس إجراءات
هل تصف الفئة بنيات التنفيذ؟ احذف نموذج النطاق مستقل عن التنفيذ

⚠️ قاعدة حرجة: البيانات الدائمة مقابل العابرة

يجب أن تمثل الفئة بيانات دائمة، وليست بيانات عابرة.

دائم: البيانات التي تحتاج إلى تخزينها في ملفات أو قاعدة بيانات (مثل: العميل، الفيلم، الطلب)

عابر: بيانات مؤقتة أثناء التنفيذ (مثل: نتائج البحث، حالة التحقق)

تقييم الارتباطات

🔗 قائمة مراجعة تقييم الارتباطات

السؤال الإجراء إذا كانت الإجابة نعم السبب
هل الارتباط غير ذي صلة بنموذج النطاق؟ احذف قم بتضمين العلاقات الضرورية فقط
هل يصف الارتباط عملية؟ احذف يجب أن يصف خاصية هيكلية، وليس حدثًا عابرًا
هل يمكن تحليل الارتباطات الثلاثية إلى ثنائية؟ حلل اجعل النموذج بسيطًا قدر الإمكان
هل هو ارتباط مشتق؟ احذف يقدم تكرارًا - يمكن حسابه من ارتباطات أخرى
هل يصف بنيات التنفيذ؟ احذف نموذج النطاق مستقل عن التنفيذ

📝 مثال: الارتباط المشتق

    A ────X──── B ────Y──── C
    
    إذا كان الارتباط X يعني "A يرتبط بـ B"
    والارتباط Y يعني "B يرتبط بـ C"
    
    إذن يمكننا اشتقاق: A يرتبط بـ C (من خلال B)
    
    لذلك، إذا كان هناك ارتباط Z: A ────Z──── C
    فهو ارتباط مشتق ويجب حذفه!

لماذا؟ لأن Z زائد عن الحاجة - يمكننا إيجاد العلاقة بين A و C بمتابعة X ثم Y.

تقييم السمات

📊 قائمة مراجعة تقييم السمات

السؤال ما يجب التحقق منه السبب
هل السمات مرتبطة ارتباطًا وثيقًا بفئتها؟ يجب أن تنتمي كل سمة منطقيًا إلى فئتها يجب أن تكون الفئة بسيطة ومترابطة
هل يجب أن تكون أي سمات حقًا فئات؟ هل المفهوم يعتمد على الفئة؟ إذا كان مستقلاً ويحتاج إلى سمات متعددة، اجعله فئة
هل يجب أن تكون أي سمات في فئة ارتباط؟ هل تعتمد السمة على وجود الارتباط؟ الكمية المباعة تعتمد على ارتباط شراء العميل-الفيلم
هل تم تضمين معرفات الكائنات؟ تحقق من حقول ID المستخدمة للعلاقات يجب ألا تكون معرفات الكائنات في نموذج النطاق!

خطأ:

العميل
- معرف العميل
- معرف الفيلم  ← هذا معرف كائن يشير إلى الفيلم!

العميل ────يشتري───→ الفيلم

5. دراسة حالة محل الأفلام الكاملة

بيان المتطلبات

🎬 متطلبات نظام محل الأفلام

فيما يلي متطلبات نظام قائم على الويب لأتمتة إدارة بيع وتأجير الأفلام لمحل أفلام:

  1. يجب أن يكون النظام قادرًا على التعامل مع الأفلام المادية والرقمية.
  2. يجب أن يكون قادرًا على تسجيل أي أفلام تُباع وتُؤجر ومن قبل من.
  3. بالنسبة للأفلام المباعة، يجب تسجيل الكمية المباعة؛ بالنسبة لتأجير الأفلام المادية، يجب تسجيل أي نسخة تم تأجيرها ومتى يجب إرجاعها.
  4. يجب أن يتتبع النظام التأجيرات المتأخرة للأفلام المادية وإرسال إشعارات بالبريد الإلكتروني للعملاء الذين لديهم أفلام متأخرة.
  5. سيكون هناك خيار عضوية العميل مقابل رسوم سنوية، مما يؤهل العضو للحصول على خصومات (10٪) على بيع وتأجير الأفلام.
  6. يجب أن يكون الأعضاء قادرين على إجراء حجوزات لتأجير الأفلام المادية إما شخصيًا في المحل، أو عبر الهاتف أو عبر الويب.
  7. يمكن للعضو حجز أكثر من خمسة أفلام مادية في أي وقت واحد، ولكن لا يوجد حد على عدد الأفلام المادية التي يمكن للعضو أو غير العضو تأجيرها في أي وقت.
  8. كخاصية مضافة، يرغب المحل في السماح للعملاء (سواء أعضاء أو غير أعضاء) بإدخال، عبر الويب، تقييمات مصغرة (حتى 100 كلمة) وتقييم (من 1، الأدنى، إلى 10، الأعلى) للأفلام التي اشتروها أو أجروها.
  9. يجب أن تكون هذه التقييمات مجهولة إذا رغب العميل في ذلك (أي يمكن للعملاء تحديد ما إذا كانوا يريدون الإفصاح عن أسمائهم عند تصفح العملاء الآخرين للتقييمات).
  10. يجب أن يكون كاتب المبيعات قادرًا على إدخال وتحديث المعلومات التالية حول جميع العملاء (أعضاء أو غير أعضاء): الاسم، العنوان، رقم الهاتف، العمر، الجنس، وعنوان البريد الإلكتروني.
  11. يتم تعيين رقم عضوية للأعضاء من قبل المحل عندما يصبحون أعضاء وكلمة مرور، مما يسمح لهم بتغيير معلوماتهم الشخصية وشراء وتأجير الأفلام الرقمية عبر الويب.
  12. يجب أن يكون مدير المحل قادرًا على إنشاء تقارير متنوعة حول بيع وتأجير الأفلام.
  13. يجب أن يكون كاتب المبيعات قادرًا على بيع وتأجير الأفلام المادية ومعالجة إرجاع الأفلام المؤجرة المادية.
  14. عند بيع أو تأجير الأفلام المادية، يجب أن يكون كاتب المبيعات قادرًا على البحث عن معلومات العميل وتحديد ما إذا كان العميل عضوًا.
  15. يجب أن يكون كاتب المبيعات قادرًا على إدخال المعلومات الأساسية عن الفيلم (أي معرف الفيلم، العنوان، الممثل الرئيسي (الممثلون)، المخرج، المنتج، النوع، الملخص، سنة الإصدار، مدة العرض، سعر البيع، وسعر التأجير).

تحليل متطلبات مفصل

🔍 عملية التحليل خطوة بخطوة

نحلل كل متطلب لاستخراج الفئات، السمات، الارتباطات، والقيود:

المتطلب 1: الأفلام المادية والرقمية

"يجب أن يكون النظام قادرًا على التعامل مع الأفلام المادية والرقمية."

التحليل:

  • الفئات: الفيلم
  • السمات:
    • الفيلم: مادي (منطقي)
    • الفيلم: رقمي (منطقي)

المنطق: يمكن أن يكون الفيلم ماديًا ورقميًا، لذلك نستخدم علامات منطقية بدلاً من إنشاء فئات فرعية منفصلة.

المتطلب 2: تسجيل المبيعات والتأجيرات

"يجب أن يكون قادرًا على تسجيل أي أفلام تُباع وتُؤجر ومن قبل من."

التحليل:

  • الفئات والارتباطات:
    • العميل يشتري الفيلم
    • العميل يؤجر الفيلم

المنطق: "من قبل من" تشير إلى فئة العميل، والأفعال "تباع" و"تؤجر" تنشئ ارتباطات.

المتطلب 3: الكمية وتفاصيل النسخة

"بالنسبة للأفلام المباعة، يجب تسجيل الكمية المباعة؛ بالنسبة لتأجير الأفلام المادية، يجب تسجيل أي نسخة تم تأجيرها ومتى يجب إرجاعها."

التحليل:

  • الفئات: نسخة تأجير
  • الارتباطات:
    • الفيلم له مادي نسخة تأجير
    • العميل يؤجر مادي نسخة تأجير
  • السمات:
    • المشتريات: الكمية (عدد صحيح) - سمة فئة الارتباط
    • نسخة تأجير: رقم النسخة (سلسلة)
    • نسخة تأجير: تاريخ الإرجاع (تاريخ)

المنطق: نحتاج إلى فئة نسخة تأجير لتتبع النسخ المادية الفردية. الكمية تنتمي إلى ارتباط الشراء.

المتطلب 4: تتبع التأخير

"يجب أن يتتبع النظام التأجيرات المتأخرة للأفلام المادية وإرسال إشعارات بالبريد الإلكتروني للعملاء الذين لديهم أفلام متأخرة."

التحليل:

  • السمات: العميل: البريد الإلكتروني (سلسلة)

المنطق: البريد الإلكتروني مطلوب للإشعار. يمكن اشتقاق حالة التأخير من تاريخ الإرجاع.

المتطلب 5: العضوية والخصومات

"سيكون هناك خيار عضوية العميل مقابل رسوم سنوية، مما يؤهل العضو للحصول على خصومات (10٪) على بيع وتأجير الأفلام."

التحليل:

  • التعميم: العضو هو نوع من العميل
    • العميل يعمم العضو

المنطق: العضو هو نوع متخصص من العميل بسمات وامتيازات إضافية.

المتطلب 6: الحجوزات

"يجب أن يكون الأعضاء قادرين على إجراء حجوزات لتأجير الأفلام المادية إما شخصيًا في المحل، أو عبر الهاتف أو عبر الويب."

التحليل:

  • الارتباط: العضو يحجز نسخة تأجير

المنطق: فقط الأعضاء يمكنهم إجراء الحجوزات. الطريقة (شخصيًا، هاتف، ويب) ليست جزءًا من نموذج النطاق.

المتطلب 7: حدود الحجز والتأجير

"يمكن للعضو حجز أكثر من خمسة أفلام مادية في أي وقت واحد، ولكن لا يوجد حد على عدد الأفلام المادية التي يمكن للعضو أو غير العضو تأجيرها في أي وقت."

التحليل:

  • القيود:
    • أقصى-كارد(العضو، يحجز) = 5
    • أقصى-كارد(العميل، يؤجر مادي) = *

المنطق: قيود التعددية تلتقط قواعد الأعمال حول حدود الكمية.

المتطلب 8: التقييمات والتصنيفات

"كخاصية مضافة، يرغب المحل في السماح للعملاء (سواء أعضاء أو غير أعضاء) بإدخال، عبر الويب، تقييمات مصغرة (حتى 100 كلمة) وتقييم (من 1، الأدنى، إلى 10، الأعلى) للأفلام التي اشتروها أو أجروها."

التحليل:

  • الفئات: التقييم
  • الارتباطات:
    • العميل يقدم التقييم
    • التقييم خاص بـ الفيلم
  • السمات:
    • التقييم: نص التقييم (سلسلة)
    • التقييم: التصنيف (عدد صحيح)

المنطق: التقييم هو فئة منفصلة لأنه يحتوي على سمات متعددة ويربط فئتين أخريين.

المتطلب 9: التقييمات المجهولة

"يجب أن تكون هذه التقييمات مجهولة إذا رغب العميل في ذلك (أي يمكن للعملاء تحديد ما إذا كانوا يريدون الإفصاح عن أسمائهم عند تصفح العملاء الآخرين للتقييمات)."

التحليل:

  • السمات: التقييم: مجهول (منطقي)

المنطق: علامة للإشارة إلى ما إذا كان يجب عرض اسم العميل في التقييم.

المتطلب 10: معلومات العميل

"يجب أن يكون كاتب المبيعات قادرًا على إدخال وتحديث المعلومات التالية حول جميع العملاء (أعضاء أو غير أعضاء): الاسم، العنوان، رقم الهاتف، العمر، الجنس، وعنوان البريد الإلكتروني."

التحليل:

  • السمات:
    • العميل: الاسم (سلسلة)
    • العميل: العنوان (سلسلة)
    • العميل: رقم الهاتف (سلسلة)
    • العميل: العمر (عدد صحيح)
    • العميل: الجنس (سلسلة)
    • العميل: البريد الإلكتروني (سلسلة)

المنطق: تعيين مباشر من المتطلبات إلى السمات.

المتطلب 11: بيانات اعتماد العضو

"يتم تعيين رقم عضوية للأعضاء من قبل المحل عندما يصبحون أعضاء وكلمة مرور، مما يسمح لهم بتغيير معلوماتهم الشخصية وشراء وتأجير الأفلام الرقمية عبر الويب."

التحليل:

  • السمات:
    • العضو: رقم العضوية (سلسلة)
    • العضو: كلمة المرور (سلسلة)

المنطق: هذه السمات خاصة بالعضو، وليس جميع العملاء.

المتطلبات 12-14: المتطلبات الوظيفية

"يجب أن يكون مدير المحل قادرًا على إنشاء تقارير متنوعة..."

"يجب أن يكون كاتب المبيعات قادرًا على بيع وتأجير الأفلام المادية..."

"عند بيع أو تأجير الأفلام المادية، يجب أن يكون كاتب المبيعات قادرًا على البحث عن معلومات العميل..."

التحليل:

  • لا توجد متطلبات جديدة لنموذج النطاق

المنطق: هذه متطلبات وظيفية تصف ما يمكن للمستخدمين فعله، وليست بيانات تحتاج إلى تخزين.

المتطلب 15: معلومات الفيلم

"يجب أن يكون كاتب المبيعات قادرًا على إدخال المعلومات الأساسية عن الفيلم (أي معرف الفيلم، العنوان، الممثل الرئيسي (الممثلون)، المخرج، المنتج، النوع، الملخص، سنة الإصدار، مدة العرض، سعر البيع، وسعر التأجير)."

التحليل:

  • السمات:
    • الفيلم: معرف الفيلم (سلسلة)
    • الفيلم: العنوان (سلسلة)
    • الفيلم: الممثل الرئيسي[0..*] (سلسلة) - إمكانية وجود ممثلين رئيسيين متعددين
    • الفيلم: المخرج (سلسلة)
    • الفيلم: المنتج (سلسلة)
    • الفيلم: النوع (سلسلة)
    • الفيلم: الملخص (سلسلة)
    • الفيلم: سنة الإصدار (عدد صحيح)
    • الفيلم: مدة العرض (عدد صحيح)
    • الفيلم: سعر البيع (حقيقي)
    • الفيلم: سعر التأجير (حقيقي)

المنطق: لاحظ أن الممثل الرئيسي له تعددية [0..*] لأنه يمكن أن يكون للفيلم ممثلون رئيسيون متعددون.

الحل 1: ارتباطات مباشرة

         (الكمية)
         
العميل ─────يؤجر────→ الفيلم

القيود: إذا كانت المشتريات فئة ارتباط، 
فيمكن للعميل شراء فيلم معين مرة واحدة على الأكثر.

الإيجابيات: بسيط، مباشر

السلبيات: لا يمكن تتبع مشتريات متعددة لنفس الفيلم، لا يمكن تتبع سجل التأجير

الحل 2: فئات المعاملات (مفضل)

العميل ─────لديه────→ شراء ─────له شراء────→ الفيلم
                       (الكمية)

العميل ─────لديه────→ تأجير ─────له تأجير────→ الفيلم

الميزة: يمكن تتبع معاملات متعددة مع مرور الوقت
يمكن تخزين تواريخ المعاملات، المبالغ، إلخ.

الإيجابيات: أكثر مرونة، يتتبع التاريخ، يسمح بمشتريات متعددة

السلبيات: أكثر تعقيدًا

السلبيات: الأكثر تعقيدًا، قد يكون مبالغًا فيه

قيود التعددية التفصيلية

🔢 تحديد التعدديات

يتم تحديد التعدديات من:

  • المتطلبات الصريحة - المذكورة في المتطلبات
  • المعرفة بمجال العالم الحقيقي - المنطق السليم حول المجال
  • غير معروف ("؟") - عندما لا يمكن الاستدلال

من المعرفة بالعالم الحقيقي

الارتباط الحد الأدنى الحد الأقصى المنطق
العضو يحجز نسخة تأجير 0 5 قد لا يكون للعضو حجوزات؛ الحد الأقصى 5 من المتطلبات
نسخة تأجير يحجزها عضو 0 1 قد لا تكون النسخة محجوزة؛ يمكن حجزها فقط من قبل عضو واحد

الفيلم له تأجير تأجير 0 * قد لا يتم تأجير الفيلم أبدًا؛ يمكن أن يكون له العديد من التأجيرات
التأجير له تأجير فيلم 1 1 كل تأجير هو لفيلم واحد بالضبط

الفيلم مقابل النسخة المادية

مثيل فئة الفيلم لا يمثل مثيل نسخة مادية من الفيلم، بل يمثل وصف الفيلم.

وبالتالي، يمكن أن يرتبط نفس مثيل الفيلم (أي الوصف) بالعديد من نسخ التأجير، والتأجيرات، والمشتريات.

6. أخطاء نمذجة النطاق الشائعة

❌ أخطاء شائعة يجب تجنبها

1. استخدام المعرفات لربط الفئات

خطأ:

العميل
- معرف العميل
- معرف الفيلم  ← لا تستخدم المعرفات لربط الفئات!

صحيح:

العميل ────ارتباط───→ الفيلم

لماذا؟ يجب ألا تتضمن معرفات الكائنات كسمات في نموذج النطاق! استخدم الارتباطات بدلاً من ذلك.

2. تمثيل العميل/المنظمة

خطأ:

┌──────────┐
│   المحل   │
└──────────┘

لماذا؟ "المحل" هو المنظمة التي تدير النظام، وليست بيانات تحتاج إلى تخزين. لا تضعه في نموذج النطاق.

3. استخدام غير صحيح للتعميم

خطأ:

      شخص
        △
        │
    ┌───┴────┐
عميل  غير عضو

فيلم مادي  فيلم رقمي
        △          △
        └────┬─────┘
           فيلم

لماذا؟

  • شخص عام جدًا - غير مطلوب
  • غير عضو هو مجرد عميل بدون عضوية - لا حاجة لفئة منفصلة
  • فيلم مادي/فيلم رقمي - يمكن أن يكون الفيلم كليهما، لذا استخدم سمات بدلاً من ذلك

صحيح:

    عميل
        △
        │
      عضو

فيلم (بسمات مادي، رقمي)

4. استخدام غير صحيح للتجميع/التركيب

خطأ:

فيلم ◇────── نسخة تأجير
فيلم ◇────── تقييم
عميل ◇────── عضو

لماذا؟ هذه ليست علاقات جزء-كل!

  • نسخة تأجير ليست "جزءًا" من الفيلم - إنها مثيل مادي منفصل
  • التقييم ليس "جزءًا" من الفيلم - إنها كيان مستقل
  • العضو ليس "جزءًا" من العميل - إنه نوع متخصص (استخدم التعميم)

5. استخدام غير صحيح لفئة الارتباط

خطأ:

العميل ─────يؤجر────→ الفيلم
           (تاريخ الإرجاع)

المشكلة: تاريخ الإرجاع لنسخة محددة، وليس الفيلم بشكل عام!

العميل ─────يؤجر مادي────→ نسخة تأجير

10. المبالغة في تحديد النموذج

خطأ:

┌──────────────┐
│ كاتب المبيعات   │  ← دور، وليس بيانات دائمة
│   المدير    │  ← دور، وليس بيانات دائمة
└──────────────┘

لماذا؟ هذه أدوار مستخدم للمتطلبات الوظيفية، وليست بيانات لتخزينها.

سؤال رئيسي: ما المعلومات حول الأشياء/الإجراءات التي تحتاج إلى أن تكون دائمة (أي في ملفات أو قاعدة بيانات)؟

💡 القاعدة الذهبية لنمذجة النطاق

قم بتضمين الفئات والارتباطات والسمات التي تمثل البيانات الدائمة التي يحتاجها النظام فقط.

اسأل نفسك: "هل تحتاج هذه المعلومات إلى تخزينها في قاعدة بيانات؟" إذا لا، فهي ربما لا يجب أن تكون في نموذج النطاق.

7. نصائح واستراتيجية الامتحان

📝 كيفية التعامل مع أسئلة نمذجة النطاق

عملية خطوة بخطوة

  1. 1اقرأ المتطلبات بعناية - عدة مرات إذا لزم الأمر
  2. 2حدد جميع الأسماء → فئات محتملة
  3. 3حدد جميع الأفعال → ارتباطات محتملة
  4. 4ابحث عن عبارات الملكية → سمات محتملة
  5. 5طبق معايير التقييم لتصفية العناصر غير الصالحة
  6. 6حدد التعدديات من المتطلبات ومعرفة المجال
  7. 7تحقق من التعميمات (علاقات هو/هي)
  8. 8حدد فئات الارتباط (السمات التي تعتمد على الارتباطات)
  9. 9ارسم المخطط بوضوح ومقروئية
  10. 10راجع باستخدام قائمة مراجعة الأخطاء الشائعة

إدارة الوقت

  • مرحلة التحليل (40٪): ضع قائمة بالفئات، الارتباطات، السمات، القيود
  • مرحلة التقييم (20٪): طبق معايير التقييم، أزل العناصر غير الصالحة
  • مرحلة الرسم (30٪): ارسم مخطط الفئات
  • مرحلة المراجعة (10٪): تحقق من الأخطاء، تحقق من التعدديات

أنواع أسئلة الامتحان الشائعة

النوع 1: نموذج النطاق الكامل

"من بيان المتطلبات، حدد جميع الفئات، السمات، الارتباطات، التجميعات/التركيبات، التعميمات، وقيود التعددية. أنشئ مخطط فئات كامل."

ما يجب فعله:

  • أنشئ جدول تحليل شامل
  • ضع قائمة بجميع العناصر بشكل منهجي
  • ارسم مخططًا كاملاً بجميع التعدديات
  • تضمن جميع السمات بأنواعها

النوع 2: تحديد عناصر محددة

"ضع قائمة بجميع الفئات التي يجب تضمينها في نموذج النطاق."

ما يجب فعله:

  • ركز فقط على ما يُسأل
  • طبق معايير التقييم بدقة
  • برر سبب تضمين كل عنصر

النوع 3: إيجاد الأخطاء

"تم إنشاء نموذج النطاق التالي للنظام. حدد الأخطاء واشرحها."

ما يجب فعله:

  • تحقق مقابل جميع معايير التقييم
  • ابحث عن الأخطاء الشائعة (المعرفات، العمليات، تفاصيل التنفيذ)
  • تحقق من التعدديات
  • تحقق من العلاقات المفقودة أو غير الصحيحة

مساعدات الذاكرة

🧠 ذاكري

الفئات هي أسماء:

  • أسماء في المتطلبات
  • كائنات في العالم الحقيقي
  • فريدة ومحددة جيدًا
  • تحتاج للتخزين
  • مستقرة مع مرور الوقت

الارتباطات هي أفعال:

  • أفعال تربط الفئات
  • فعاليات دائمة
  • تربط فئتين أو أكثر
  • بنائية مفضلة
  • هيكلية، وليست تشغيلية

تجنب ROID:

  • R أدوار (كاتب المبيعات، المدير)
  • O عمليات (يولد، يتصفح)
  • I تنفيذ (نظام، ويب)
  • D مشتقة (يمكن حسابها)
