Identification

Definition Identification = establishing the identity of a subject (user, program, process). Uses public info like a username.
📋 Good Identification Requirements
  • Unique — each value must be unique for user accountability
  • Standard Naming — a consistent naming scheme should be followed
  • Non-descriptive — should NOT reveal the user's position or role

Authentication — The 3 Factors

🧠 3 Factors Mnemonic
"Know. Own. Are."
Know — Something you KNOW (password, PIN, passphrase)
Own — Something you HAVE (token, smart card)
Are — Something you ARE (biometrics: fingerprint, iris)
🧠
Factor 1: Knowledge
Something you KNOW. Passwords, passphrases, PINs. Most common but weakest — can be guessed, stolen, or shared.
Type 1
💳
Factor 2: Ownership
Something you HAVE. Tokens, smart cards, key fobs. RSA SecurID is the classic example. Generates OTPs.
Type 2
🧬
Factor 3: Characteristics
Something you ARE. Biometrics — digitized physical features (fingerprint, iris) or behaviors (signature, keystroke).
Type 3
🔒
MFA — Multi-Factor Auth
Using 2 or more factors together = strong authentication. Must combine DIFFERENT types (e.g., password + token = 1+2). Two of the same type = NOT MFA.
Best Practice

Biometric Error Rates — The Key Exam Concept

🧠 Error Types Mnemonic
"Type 1 = Rejects the Right one. Type 2 = Accepts the Wrong one."
Type II (False Acceptance) is the MORE DANGEROUS error — you don't want impostors getting in!
Type I Error
🚫✅
False Rejection Rate (FRR)
Rejects an AUTHORIZED user. Annoying but not dangerous. System is too strict.
Type II Error
✅🚫
False Acceptance Rate (FAR)
Accepts an UNAUTHORIZED user. Very dangerous! System is too lax. The error to AVOID.
CER / ERR
⚖️
Crossover Error Rate
Point where FRR = FAR. Lower CER = More accurate system. Used to compare biometric systems.
⚠️ Exam Trap: Which error is worse? Type II (False Acceptance Rate) is always the more critical error to minimize — accepting an impostor is a security breach. Type I just inconveniences a legitimate user.

Electronic Monitoring
Sniffing network traffic to capture passwords in transit. Defeated by encryption (TLS/HTTPS).

Gaining access to the stored password file (e.g., /etc/shadow on Linux). Defeated by hashing and salting.

Trying every possible combination. Takes time but guarantees success. Defeated by lockout policies and long passwords.
Exhaustive
📖
Dictionary Attack
Uses a list of common words and passwords. Much faster than brute force. Defeated by random/complex passwords.
Wordlist

One-Time Passwords (OTP)

✅ OTP Advantage OTPs (Dynamic Passwords) are NOT vulnerable to electronic eavesdropping, sniffing, or password guessing — because each password is used exactly once.
OTP Type How it works Example
Synchronous (Time-Based) Token generates OTP in sync with the server clock. No challenge needed — just match the current time window. Google Authenticator, RSA SecurID
Asynchronous (Challenge-Based) Server sends a random challenge value → user enters it into the token → token generates OTP → user sends OTP back to server. Challenge-Response tokens

Authorization

Definition Authorization = determining whether a proven identity has the RIGHT to access requested resources. Based on level of trust + need to know.

Critical Authorization Concepts

🦠
Authorization Creep
When employees accumulate permissions over time without old ones being removed. A major security risk — regularly review and revoke unneeded access.
Risk
0️⃣
Default to Zero
Start with ZERO access and only grant what is needed. Opposite of "allow by default." This is the secure baseline principle.
Principle
🎯
Need to Know
Users get access ONLY to information they absolutely require for their job duties. Even if they have a clearance, they must have a need to know.
Principle
📋
Access Control Lists (ACL)
A list of subjects (users) that are authorized to access a particular object (resource). Bound to the OBJECT. Tells the object who can access it.
Tool
🔗
Single Sign-On (SSO)
One login for everything — benefits, risks, and technologies
What is SSO? Single Sign-On allows user credentials to be entered ONE time, granting access to ALL authorized resources across primary and secondary network domains. Also called "reduced sign-on." Used in federated identity management systems.
🧠 SSO Technologies Mnemonic
"Kerberos Sits Securely, Drawing Diagrams"
Kerberos
Sesame
Security Domains
Directory Services
Dumb Terminals
⚠️ SSO Cons
Single Point of Compromise — one stolen credential = access to everything
Legacy Interoperability — older systems may not work with SSO software
Implementation Difficulties — unusual systems may not interface well
✅ SSO Pros
Efficient logon — user logs in only once for all authorized systems
Stronger passwords — one password to remember → users choose harder ones
Centralized administration — consistent policy application across the org
⚠️ Key SSO Risk — Single Point of Failure The biggest downside: one compromised login = attacker accesses ALL of the user's authorized resources. This is the critical exam trade-off for SSO.

Access Control Models
DAC, MAC, and RBAC — know them cold
🧠 3 Models Mnemonic
"Decide by: Discretion, Mandatory labels, or Roles"
DAC — Owner decides (Discretionary)
MAC — System labels decide (Mandatory)
RBAC — Job role decides (Non-Discretionary)
Model Full Name Who Controls Access? Best Used When Key Trait
DAC Discretionary AC The OWNER of the resource decides who can access it Flexibility needed; small environments Most flexible, least secure
MAC Mandatory AC The SYSTEM enforces based on security labels & clearances Military/government where confidentiality is critical Most rigid, most secure
RBAC Role-Based AC Centrally-administered ROLES determine access Organizations with high staff turnover Best for orgs — easy to manage
👑
DAC — Discretionary
The OWNER of the resource decides who gets access. Like owning a file and deciding who can read it. Used in most OS file systems (Windows, Linux). Most flexible but can lead to inconsistent security.
Owner Controls
🏷
MAC — Mandatory
Uses a SECURITY LABELING SYSTEM. Users have clearances; resources have classification labels (Top Secret, Secret, etc.). System enforces — users cannot override. Used in the military.
System Controls
👔
RBAC — Role-Based
Access determined by JOB ROLE, not individual identity. Centrally administered. Best for high-turnover organizations — change the role, permissions follow automatically.
Role Controls

Rule-based, constrained interfaces, and access control matrix
🧠 3 Techniques Mnemonic
"Rules Constrain Matrices"
Rule-Based Access Control

📏
Rule-Based AC
Uses specific rules defining what CAN and CANNOT happen between subject and object. Not necessarily identity-based. Classic example: FIREWALLS. Traditionally used in MAC systems.
Firewall = Rule-Based
🚧
Constrained User Interfaces
Restricts user access by hiding or disabling options they're not allowed to use. Three types: Menus & Shells, Database Views, Physically Constrained Interfaces (e.g., ATM keypad).
UI Restriction
📊
Access Control Matrix
A table mapping subjects × objects showing what actions each subject can take on each object. Two views: Capability Table (bound to subject) and ACL (bound to object).
Subject × Object Table

Access Control Matrix — The Two Views

View Type Bound To How to Think About It
Capability Table SUBJECT (User) Read down a ROW — shows everything THIS USER can access. "Here are Alice's capabilities."
Access Control List (ACL) OBJECT (Resource) Read down a COLUMN — shows everyone who can access THIS RESOURCE. "Here's who can access File1."

Centralized vs Decentralized Administration

✅ Centralized
One entity oversees all corporate resource access
Consistent & uniform method of controlling access rights
Protocols used: RADIUS, TACACS+, Diameter
⚠️ Decentralized
Control given to people closer to the resources
No consistent control — lacks proper consistency
Harder to enforce organization-wide policy

RADIUS, TACACS & Diameter
AAA protocols — know the differences cold
🧠 RADIUS vs TACACS+ Key Differences Mnemonic
RADIUS = "Rapid but Partial" · TACACS+ = "Total Armored Control"
RADIUS → UDP, encrypts password only, combines auth+authz
TACACS+ → TCP, encrypts everything, separates auth+authz+accounting
Feature RADIUS TACACS+ Diameter
Full Name Remote Authentication Dial-In User Service Terminal Access Controller Access Control System Diameter (IETF RFC 3588)
Transport Protocol UDP (ports 1812, 1813) TCP (port 49) TCP/SCTP
Encryption Only encrypts the password field Encrypts the entire packet (except TACACS header) Full encryption
AAA Combines Authentication + Authorization (together) Separates Auth, Authz, and Accounting (true AAA) True AAA, extensible
Best For Simple username/password auth; ISP dial-up Environments needing full security & granular control Modern apps needing extensible AAA
Who Uses It Most ISPs — authenticates customers before internet access Enterprise networks — Cisco AAA architecture Mobile networks, modern AAA apps
💡 RADIUS Expanded: R-A-D-I-U-S Remote Authentication Dial-In User Service — open protocol, client/server architecture, UDP-based, only encrypts password. Appropriate for simple accept/deny decisions.
💡 TACACS+ Advantage TACACS+ separates authentication from authorization so they can be implemented on DIFFERENT security servers. Provides true AAA architecture and encrypts all data — superior to RADIUS for enterprise security.
💡 Diameter = RADIUS 2.0 Diameter was built to overcome RADIUS limitations. It's an IETF standard (RFC 3588). Applications requiring AAA can define extensions on top of the Diameter base protocol.
